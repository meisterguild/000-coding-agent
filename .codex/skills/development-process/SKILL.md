# Development Process

## Overview
プロジェクトを進めるための基本フローを定義する。
各ステップを順番に実行すること。
ただし、次のステップに進むには成果物に対する指示者の承認が絶対に必要。
成果物が作成出来たら次のステップに進まず、指示者に成果物を提出し、承認を得ること。

## Rules

- 各ステップが完了したら、必ずその成果物を提示し、「レビュー依頼」を出して作業を停止すること。
- レビューで「承認」と明示されるまで、次のステップに進んではならない。
- 承認がない状態で次工程に進むことは禁止。
- 自律的に最後まで進めないこと。
- 必ず人間の確認待ち状態で止まること。

## Steps
1. 要件定義
   - 目的、期待成果、制約を明確化する。
   - 成果物：docs/requirements.md
2. 工数見積もり
   - 各タスクの工数を見積もる。
   - 成果物：docs/estimation.md
3. スコープ定義
   - 見積もった工数が3人日を超える場合、超えない範囲になるように今回のプロジェクトのスコープを調整する。
   - 成果物：docs/scope.md
   3.1 スコープ調整
      - 工数が3人日を超える場合、要件を削減するなどしてスコープを調整し、工数を3人日以下にした要件を定義する。
      - 成果物：docs/scoped_requirements.md
4. 設計
   - 要件を満たすための設計を行う。スコープ調整を行った場合は、そちらの要件に従う。
   4.1. 外部設計
      - ユーザーインターフェースやAPIの仕様を定義する。
      - 成果物：docs/interface_design.md
   4.2. 内部設計
      - システムの内部構造やアルゴリズムの設計を行う。
      - 成果物：docs/internal_design.md
5. タスク分割
   - 設計を実行可能な単位に分解し、優先順位を決める。
   - 成果物：docs/tasks.md
6. 実装
   - タスクに沿って実装を行い、必要に応じて中間確認を行う。
   - 成果物：src/
7. 動作確認
   - 要件を満たしているか検証し、問題があれば修正する。
   - 成果物：docs/test_results.md
